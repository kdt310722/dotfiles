# dotfiles

Personal dotfiles for macOS and Linux — managed with [antidote](https://getantidote.github.io/) and [Starship](https://starship.rs/).

## Structure

```
dotfiles/
├── .zshrc              # Entry point — sets env vars, sources all zsh configs
├── .env                # Local secrets/overrides (git-ignored)
├── .zsh/
│   ├── helpers.zsh     # Utility functions (create_dir, source_if_exists, ...)
│   ├── updater.zsh     # Auto-update scheduler (dotfiles-first strategy)
│   ├── antidote.zsh    # Plugin manager setup
│   ├── aliases.zsh     # Shell aliases
│   ├── nvm.zsh         # Node version manager config
│   ├── eza.zsh         # eza (modern ls) config
│   ├── starship.zsh    # Starship prompt init
│   ├── tealdeer.zsh    # tldr client config
│   ├── direnv.zsh      # direnv hook
│   ├── completions.zsh # Completion setup
│   └── misc.zsh        # Miscellaneous settings
└── install-tools.sh    # Cross-platform tool installer (macOS/Ubuntu/Debian/Fedora/Arch)
```

## Installation

```sh
git clone git@github.com:kdt310722/dotfiles.git ~/dotfiles
echo 'ZDOTDIR=$HOME/dotfiles' >> ~/.zshenv
exec zsh
```

On first shell, missing tools are not installed automatically — run the installer manually:

```sh
~/dotfiles/install-tools.sh
```

## Auto-update

Updates run in the background once per day (configurable via `ZSH_UPDATE_INTERVAL`).

**Strategy:** dotfiles itself is checked first. If there are new remote commits, they are pulled and all other updates (plugins, tools) are deferred to the next session — ensuring config changes take effect before anything else runs.

Commands that run on update are defined in `.zshrc`:

```zsh
export UPDATE_COMMANDS=(
  "antidote update"
  "update_starship"
  "update_tealdeer"
  "update_nvm_and_node"
)
```

## Configuration

| Variable | Default | Description |
|---|---|---|
| `ZSH_UPDATE_INTERVAL` | `1` | Days between auto-updates |
| `ZSH_PLUGINS` | see `.zshrc` | antidote plugin list |
| `UPDATE_COMMANDS` | see `.zshrc` | Commands run on auto-update |
| `PROJECT_PATHS` | `~/Projects` | Paths used by the `pj` plugin |

### zsh-ai

The [matheusml/zsh-ai](https://github.com/matheusml/zsh-ai) plugin lets you type `# describe what you want` and get a shell command generated by AI. It is loaded **only when `ZSH_AI_PROVIDER` is set** in `.env`.

**Enable the plugin** by setting a provider in `.env`:

```sh
ZSH_AI_PROVIDER=anthropic   # anthropic | openai | gemini | ollama | mistral | grok
```

**API keys** (set in `.env`, one per provider in use):

| Provider | API key variable |
|---|---|
| Anthropic | `ANTHROPIC_API_KEY` |
| OpenAI | `OPENAI_API_KEY` |
| Gemini | `GEMINI_API_KEY` |
| Mistral | `MISTRAL_API_KEY` |
| Grok | `XAI_API_KEY` |
| Ollama | — (local, no key needed) |

**Per-provider model and URL overrides:**

| Variable | Default | Description |
|---|---|---|
| `ZSH_AI_ANTHROPIC_MODEL` | `claude-haiku-4-5` | Anthropic model |
| `ZSH_AI_ANTHROPIC_URL` | `https://api.anthropic.com/v1/messages` | Anthropic endpoint |
| `ZSH_AI_OPENAI_MODEL` | `gpt-4o` | OpenAI model |
| `ZSH_AI_OPENAI_URL` | `https://api.openai.com/v1/chat/completions` | OpenAI endpoint (supports LiteLLM/proxies) |
| `ZSH_AI_OPENAI_API_KEY` | — | Override key for OpenAI-compatible proxies (takes precedence over `OPENAI_API_KEY`) |
| `ZSH_AI_GEMINI_MODEL` | `gemini-2.5-flash` | Gemini model |
| `ZSH_AI_OLLAMA_MODEL` | `llama3.2` | Ollama model |
| `ZSH_AI_OLLAMA_URL` | `http://localhost:11434` | Ollama server URL |
| `ZSH_AI_MISTRAL_MODEL` | `mistral-small-latest` | Mistral model |
| `ZSH_AI_MISTRAL_URL` | `https://api.mistral.ai/v1/chat/completions` | Mistral endpoint |
| `ZSH_AI_GROK_MODEL` | `grok-4-1-fast-non-reasoning` | Grok model |
| `ZSH_AI_GROK_URL` | `https://api.x.ai/v1/chat/completions` | Grok endpoint |
| `ZSH_AI_PROMPT_EXTEND` | — | Extra instructions appended to the AI prompt |

Example `.env` for Anthropic with a custom prompt preference:

```sh
ZSH_AI_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
ZSH_AI_PROMPT_EXTEND="Always prefer ripgrep over grep, fd over find, bat over cat."
```
